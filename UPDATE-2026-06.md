## 3. NSSM 服务无法启动 —— 已修复主因 + 排查步骤

**根本原因（已在代码修复）**：`server.py` 启动时会打印含 `✓ ⚠ ✗ ✉` 和方框字符的状态行。
作为 Windows 服务由 NSSM 托管时没有 UTF-8 控制台，stdout 默认用系统代码页（中文 Windows 是 GBK），
打印第一个这种字符就会抛 `UnicodeEncodeError`，进程在启动瞬间崩溃——表现就是截图里的
"Windows 无法启动服务 / 服务并未返回错误"。

> 判别方法：如果在 cmd 窗口里 `python server.py` 能正常跑，但作为服务起不来，几乎一定就是这个编码问题。

已在 `server.py` 顶部加入 UTF-8 守卫（`sys.stdout/stderr.reconfigure(encoding='utf-8', errors='replace')`），
任何控制台下打印都不再崩溃。用新版 server.py 重装/重启服务后，这一原因即消除。

**仍起不来时的排查（强烈建议先做日志重定向，把真实错误暴露出来）：**

```bat
nssm set WFXWebsite AppStdout  D:\wfx\logs\out.log
nssm set WFXWebsite AppStderr  D:\wfx\logs\err.log
nssm restart WFXWebsite
:: 然后打开 err.log 看真实报错
```

核对这几项 NSSM 配置：
```bat
nssm set WFXWebsite Application      C:\Python311\python.exe   :: 必须是 python.exe 完整路径(或虚拟环境的)
nssm set WFXWebsite AppDirectory     D:\wfx                    :: server.py 所在目录
nssm set WFXWebsite AppParameters    server.py                 :: 云上可加 --host 0.0.0.0 --port 8000
```

**第二个可能原因**：若以生产模式运行（环境变量 `WFX_ENV=production` 或参数 `--production`），
而 `WFX_SESSION_SECRET` 没设置或仍是占位符，server.py 会**故意拒绝启动并 `sys.exit(1)`**（安全设计）。
解决：要么设置密钥，要么放一个 `config.py` 到 AppDirectory：
```bat
nssm set WFXWebsite AppEnvironmentExtra ^
  WFX_SESSION_SECRET=<32位以上随机串> ^
  WFX_DB_HOST=127.0.0.1 WFX_DB_USER=wfx_user WFX_DB_PASSWORD=<密码> WFX_DB_NAME=wfx_website
nssm restart WFXWebsite
```
